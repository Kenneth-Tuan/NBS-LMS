## Context

目前的請假流程，學生需要手動下載 doc 檔案、列印填寫、找老師簽名後送到教務處，流程不連貫且難以追蹤。雖然系統已經有線上請假申請的表單（`LeaveApplication.vue`）和 API，但缺少最後自動產出制式請假單 PDF 的環節，也未處理老師線上審核後的回填。

## Goals / Non-Goals

**Goals:**
- 在系統中實作基於 PDF 範本自動生成制式請假單的功能。
- 將學生線上填寫的表單資料與老師的審核結果合併，精確套印於制式請假單的固定欄位上。
- 提供學生在審核通過後下載 PDF 的按鈕，讓他們能自行交給教務處。

**Non-Goals:**
- 自動化教務處的收件流程（本階段只處理到產出 PDF 讓學生自行繳交）。
- 電子簽章的法律效力驗證（目前僅在 PDF 欄位套印老師姓名）。

## Decisions

- **PDF 生成工具選擇**: 沿用專案現有產生個人成績單的模式，使用 `pdf-lib` 和 `fontkit`。將 `.doc` 空白表單轉換為 PDF 格式放入 `public/`，透過設定檔定義每個欄位的 X, Y 座標。
  - *Rationale*: 專案中已經有成功使用的範例（`useTranscriptPdf.js`），技術堆疊一致且對開發者來說熟悉，能快速實作對位。

- **資料整合與表單擴充**: 在前端 `LeaveApplication.vue` 和 `application.js` (Store) 中新增請假節數等對應制式單的必要欄位。
  - *Rationale*: 制式請假單需要這些欄位才能完整填寫。如果後端 API (`applyLeave`) 尚未支援這些新欄位，會需要後端配合擴充 JSON payload 的 Schema，或者將額外資訊暫存於現有欄位 (例如將節數整合到 reason 中，但不建議，應正規化)。

- **老師簽名處理**: 當請假單狀態為 APPROVED 時，在教務處收執聯的「任課老師簽名」欄位印上審核老師的姓名與審核日期字樣（如："王大明 2025/12/03"）。
  - *Rationale*: 由於目前沒有實作複雜的電子簽章憑證系統，印上名字與日期已能提供基本證明，並滿足流程數位化的最小需求。

## Risks / Trade-offs

- **Risk: 座標偏移與跨裝置顯示問題** → Mitigation: 將使用固定解析度的 PDF 範本與相同的字型檔（如 `DFKai-SB.ttf`），並在測試時確認不同長度文字的排版效果，必要時加入字數截斷邏輯。
- **Risk: API 未支援新欄位** → Mitigation: 若後端暫時無法擴充，需討論是否在前端計算請假總節數（依據日期或課表），或是請後端配合修改。

## Open Questions

1. 確切的請假總節數邏輯：目前 `applyLeave` 只送出起迄日期，節數是要讓學生自己填寫數字，還是系統根據課表自動計算？
2. 空白請假單的兩聯（教務處收執聯與學生收執聯）的填寫內容差異：除了簽核欄位外，是否其餘資料都一致？
3. API 更新進度：是否需要與後端協調更新 `applyLeave` 的 payload 格式，以儲存學號、科系、請假節數等額外資訊。
